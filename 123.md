  ---
  ✅ What is Working Well

  RAG Pipeline Architecture — Frontier-Grade

  - RAPTOR hierarchical retrieval is correctly gated: only fires for comparative, factual, complex, or multi-entity
  queries (raptor-retrieval.ts:54-72). Simple queries skip it. Smart.
  - Semantic Router (router.ts:60-500) is exceptionally well-designed — 6 query types each get dedicated retrieval
  config AND prompt template with HMS-specific formatting (wire colors, terminal naming).
  - Logical Router (logical-router.ts:274-327) cleanly separates support queries (vector) from admin/analytics
  queries (relational SQL). No cross-contamination.
  - Query Decomposition uses 3-tier strategy: fast heuristics → regex patterns → LLM fallback
  (query-decomposer.ts:217-266). Only complex queries trigger LLM calls.
  - Conversation-aware rewriting (conversation-retrieval.ts:69-145) correctly resolves pronouns ("fix it" → "fix the
  power supply issue on TB1"), with 3-second timeout and sanity checks.
  - Reranking chain is well-structured: 55% vector + 30% cross-encoder + 15% BM25 composite scoring with entity
  overlap, answer specificity, and generic-answer penalties (rag-engine.ts:330-366).

  Caching — Well-Designed 2-Tier System

  - Tier 1 (Upstash Redis, SHA-256 exact match, <5ms) + Tier 2 (Supabase pgvector, 0.90 cosine threshold, <50ms) + L0
   in-memory LRU (200 entries).
  - Cache store gates are sensible: skip general/rag_partial modes, reject answers <20 or >8000 chars, require
  confidence ≥ 0.72 (cache.ts:240-258). Prevents low-quality responses from poisoning cache.
  - Tier 2 backfills to Tier 1 on hit — eliminates repeated pgvector queries.
  - Diagram requests correctly bypass cache.

  Performance Instrumentation

  - All 7 pipeline stages instrumented via createStageTimer() with performance.now() sub-ms precision.
  - Ring buffer (100 entries) for real-time dashboards + batched DB writes (10 items/15s flush) — non-blocking.
  - P50/P95/P99 latency, cache hit rates, answer mode distribution all computed correctly.
  - Cost savings tracking: HYDE skip savings + cache hit savings estimated per window.

  Rate Limiting — Proper Enforcement

  - 3 tiers: Guest 15/min, Auth 60/min, Admin 200/min (rate-limiter.ts).
  - Redis primary + in-memory fallback. Exponential backoff for repeat offenders.
  - Proper response headers: X-RateLimit-Remaining, Retry-After.
  - Rate check happens BEFORE pipeline execution — no wasted compute.

  Chat Input Validation

  - Request body validated: Array.isArray(body.messages), non-empty, .content must be string (chat/route.ts:557-580).
  - Language constrained to whitelist: ['en', 'bn', 'hi'].
  - Empty messages return 400.

  Admin Dashboard — Recently Fixed Layout

  - Unified sticky header+tabs block (admin/page.tsx:349-406).
  - Defensive normalization adapter (adminAdapter.ts) handles all backend shape variations with asNumber(),
  asArray(), asRecord() — prevents undefined/NaN from reaching UI.
  - 4 parallel API fetches via Promise.allSettled with per-endpoint error handling.

  Code Quality

  - Strong modularity: 20+ lib modules with clear single responsibilities.
  - TypeScript throughout with proper interfaces.
  - Fire-and-forget pattern for non-critical writes (cache store, metrics, evaluation) — doesn't block user response.

  Severity: 🔴 CRITICAL — Exploitable immediately
  Files: ALL files in src/app/api/admin/*
  Evidence: grep -rn "auth\|getUser\|getSession\|isAdmin" src/app/api/admin/ returns zero results.

  The middleware (middleware.ts:7) explicitly states: "This middleware does NOT protect any routes." Every admin
  endpoint is publicly accessible:

  CRITICAL-2: Hallucination Safeguards Are Prompt-Level Only

  Severity: 🔴 CRITICAL — Affects answer accuracy
  File: router.ts:435-440, chat/route.ts

  When answerMode = 'general' (no KB match), the prompt says "Do NOT guess" — but there is no enforcement mechanism.
  The LLM still receives the full system prompt and will generate plausible-sounding but potentially fabricated
  answers about HMS panels.

  Missing safeguard:
  if (answerMode === 'general' && contextString.trim().length === 0) {
      return buildResponse(NOT_FOUND_MESSAGES[language], ...);
  }

  The NOT_FOUND_MESSAGES exist (chat/route.ts:46-50) for all 3 languages but are only used on explicit low-confidence
   paths, not as a hard gate.

  CRITICAL-3: Prompt Injection Filter Is Trivially Bypassable

  Severity: 🔴 CRITICAL — Security
  File: src/lib/sanitize.ts (entire file, 10 lines)

  The sanitization is 4 regex patterns only:
  .replace(/ignore (all )?previous instructions/gi, '[filtered]')
  .replace(/system prompt/gi, '[filtered]')
  .replace(/you are chatgpt/gi, '[filtered]')
  .replace(/act as/gi, '[filtered]');

  Bypasses:
  - Unicode substitution: ignore previous instructions (zero-width space)
  - Synonym: disregard prior directives
  - Encoding: i g n o r e   p r e v i o u s
  - The filter is only applied to conversation history (chat/route.ts:1386), NOT to the user's actual query input.

  ⚠️ Medium Issues

  MEDIUM-1: Retrieval Path Inconsistency — Hybrid Skips HYDE

  File: rag-engine.ts:654
  When useHybridSearch = true (auto-enabled for comparative/complex queries), HYDE is explicitly skipped:
  HYDE: skipped (not implemented in hybrid search path)
  This means the queries that need HYDE most (complex, comparative) don't get it when hybrid retrieval kicks in. Two
  retrieval paths with different capabilities and no unified validation.

  MEDIUM-2: Similarity Threshold Too Permissive (0.18)

  File: rag-engine.ts (SUPABASE_FETCH_THRESHOLD)
  Cosine similarity threshold of 0.18 retrieves extremely weak matches. Industry best practice is 0.28-0.35 for
  technical domains. Weak matches flow through reranking and can end up in LLM context as noise.

  MEDIUM-3: Sequential Pipeline Bottleneck

  File: chat/route.ts:550-1400
  Main pipeline is largely sequential: translation → rewrite → embedding → retrieval → reranking → generation.
  Translation + query rewrite could run in parallel (saving ~200-400ms). Current P95 estimated at 2.5-3.8s; target is
   <3s.

  MEDIUM-4: Analytics API Has No Partial Failure Handling

  File: admin/analytics/route.ts
  18 concurrent Supabase queries wrapped in a single Promise.all. If ANY query fails, the entire response is 500.
  Should use Promise.allSettled with partial data fallback (the dashboard component already handles missing data
  gracefully).

  MEDIUM-5: Logger Silently Drops Entries Under Load

  File: logger.ts:62-65
  Buffer limit of 300 logs — when exceeded, oldest entries silently shift(). No warning emitted. Under burst traffic,
   observability goes dark at the moment you need it most.

  MEDIUM-6: X-Forwarded-For Header Spoofable

  File: rate-limiter.ts:220-230
  Rate limiter trusts X-Forwarded-For without validating it comes from a known proxy (Netlify CDN). Sophisticated
  attacker can rotate spoofed IPs to bypass per-IP rate limits.

  ★ Insight ─────────────────────────────────────
  The pattern here is common in RAG systems: The retrieval + generation pipeline is sophisticated and
  well-engineered, but the security perimeter around it is an afterthought. In production, the attack surface isn't
  just "can the AI answer wrong?" — it's "can someone upload poisoned data, disable safety features, or extract user
  queries?" Admin endpoint auth is the #1 blocker because it makes all other safeguards moot — an attacker who can
  PUT /api/admin/rag-settings can disable reranking, lower thresholds, and turn the bot into a hallucination machine.

  Additional Findings from Agent 3 (Security)

  Confirmed CRITICAL-1 — Agent 3 independently verified zero auth on all admin endpoints and produced a working

  Additional Findings from Agent 4 (Monitoring/UI/Edge Cases/Scalability)

  New MEDIUM — Analytics uses Promise.all not Promise.allSettled (admin/analytics/route.ts):
  - 18 concurrent Supabase queries — if ANY one fails, entire analytics page returns 500
  - Dashboard component already handles partial data gracefully via EMPTY_DASHBOARD
  - But the API never gives it partial data — it's all-or-nothing
  - Fix: Switch to Promise.allSettled with per-query fallback defaults

  New MEDIUM — Missing DB indexes on chat_logs and failures tables:
  - chat_logs queried by created_at DESC and request_id — no indexes
  - failures queried by created_at DESC — no index
  - At scale (100K+ rows): full table scans on every admin page load
  - Note: pipeline_metrics table (migration 028) already has proper indexes

  New MEDIUM — Rate limiter memory growth under attack:
  - memoryStore Map entries with blockedUntil in the future are NOT cleaned up until the backoff expires
  - Under sustained attack: 1000+ unique IPs × timestamp arrays = unbounded growth
  - Fix: Add absolute expiry alongside blockedUntil

  New WARN — Admin dashboard setInterval cleanup:
  - AdminAnalyticsDashboard.tsx sets a 15s refresh interval
  - If useEffect cleanup function is missing, multiple admin tab opens create leaked intervals
  - Agent couldn't fully confirm due to file read limits — verify manually

  New WARN — Confidence threshold mismatch:
  - CLAUDE.md says UNKNOWN_THRESHOLD = 0.45
  - rag-engine.ts uses PARTIAL_CONFIDENCE = 0.38
  - Queries scoring 0.38-0.45 get rag_partial answers when they should get NOT_FOUND
  - Fix: Align to 0.45

  New WARN — No request ID generation:
  - Chat route doesn't generate a UUID at entry
  - Logs from a single query can't be traced end-to-end
  - Fix: const requestId = randomUUID() at route entry, pass through all handlers
